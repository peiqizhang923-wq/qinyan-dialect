## 一、市场调研

### 1.1 方言保护的时代紧迫性

中国拥有十大汉语方言区，其中陕西方言作为中原官话的核心分支，是周秦汉唐雅言的活态遗存。然而，随着城镇化加速和普通话推广，方言使用场景急剧萎缩：

- 国家语委 2022 年调查显示，**20-30 岁人群中仅 38%** 能熟练使用家乡方言，较上一代下降 27 个百分点
- 陕西省 89 个区县中，**超过 60% 的青少年**日常已不使用本地方言交流
- 中国语言资源保护工程截至 2024 年已记录濒危方言点 **1,500 余个**，保护任务紧迫

### 1.2 政策驱动力

| 政策文件 | 核心要求 | 与项目关联 |
|----------|---------|-----------|
| 教育部《中国语言资源保护工程》（2015-2025） | 全国方言调查与数字化存档 | 本项目方言数字化保护看板 |
| 国务院《新一代人工智能发展规划》（2017） | AI 与文化遗产保护融合 | 大模型驱动的方言智能转译 |
| 文旅部《"十四五"文化和旅游发展规划》（2021） | 非遗数字化、沉浸式体验 | 游戏化方言文化互动设计 |
| 陕西省《陕西方言集成》（2019-2024） | 10 卷丛书、310GB 音频建档 | 项目参考的权威方言语料 |

### 1.3 竞品分析

从"方言翻译工具""非遗文化体验"维度对比现有产品：

| 维度 | 讯飞输入法方言版 | 高德方言导航 | 地方博物馆 AR | **秦言三韵** |
|------|-----------------|-------------|--------------|-----------|
| 方言种类 | 粤语/四川话/东北话等 | 少量方言包 | 无 | **关中/陕北/陕南全覆盖** |
| AI 翻译 | 词表替换 | 预录音频 | 无 | **大模型语义理解转译** |
| 语音合成 | 普通话 TTS | 录制配音 | 无 | **陕西话神经网络语音** |
| 文化体验 | 无 | 无 | 静态展示 | **翻译+解码+考试+祈福+转盘** |
| 评测体系 | 无 | 无 | 无 | **50条语料自动量化评测** |
| 部署门槛 | 需联网 | 需特定设备 | 需线下 | **浏览器即开即用** |

**市场空白**：目前在"AIGC + 方言转译 + 游戏化体验 + 非遗保护"的交叉领域，尚无面向陕西三大方言区的系统化产品。

## 二、UI 设计思路

### 2.1 设计定位：盛唐美学 × 数字交互

以唐代长安城（今西安）为核心文化锚点，将"秦砖汉瓦的大气 + 唐三彩的华彩 + 丝绸之路上交融"凝练为视觉语言。整体调性：**古雅而不陈旧，华美而不轻浮**。

### 3.2 前端技术选型

**全栈 Vanilla——零框架、零构建、零依赖：**

| 决策 | 选型 | 理由 |
|------|------|------|
| 框架 | 无（纯 HTML/CSS/JS） | 11 个独立页面，共享 common.js，无需 SPA 框架增加复杂度 |
| 构建 | 无 | 直接 `file://` 打开即可运行，零 `npm run build` |
| CSS | 原生 CSS Variables | 11 个颜色变量 + `cubic-bezier` 全局过渡 |
| 状态管理 | localStorage + `STATE` getter/setter 对象 | 跨页面状态持久化，无需 Redux/Vuex |
| 页面导航 | `window.location.href` | 多页面最简单的导航方式，配合 `fadeTo()` 过渡 |
| 字体 | 系统字体栈 | 不加载 Web Font，楷体/雅黑 Windows 系统自带 |

**设计依据**：比赛现场演示对稳定性要求极高，"少一个依赖就少一个出错可能"。纯前端方案在 file:// 下即可完整运行，无任何构建或服务端依赖（AI/TTS 除外）。

### 3.3 后端服务设计

系统包含两个轻量级 Python 微服务，均使用 Python 标准库实现，无 Flask/Django 等重量级框架。

**服务一：TTS 语音服务（`tts_server.py`，端口 9880）**

| 特性 | 实现 |
|------|------|
| 框架 | `http.server.BaseHTTPRequestHandler` + `ThreadingMixIn` 多线程 |
| 语音引擎 | Microsoft Edge-TTS `zh-CN-shaanxi-XiaoniNeural` |
| 风格映射 | 5 档语速/音调差异化参数（关中/陕北/陕南/古风/通用） |
| 容错 | 最多 3 次重试，指数退避（0.8×2^n + 随机抖动），最小文件校验（≥100 bytes） |
| 临时文件 | `tempfile` 自动清理，`finally` 块保证删除 |

**服务二：数据库 API 服务（`db_server.py`，端口 9881）**

| 特性 | 实现 |
|------|------|
| 数据库 | SQLite（WAL 模式，外键约束） |
| 表结构 | 4 张表：`history` / `eval_results` / `settings` / `dialect_vocabulary` |
| API 设计 | RESTful JSON，6 个端点，装饰器路由注册 |
| 种子数据 | 24 条方言词汇（关中10 + 陕北7 + 陕南7），含发音、释义、典故 |
| 依赖 | 零外部依赖，纯 Python 标准库（sqlite3 + http.server + json） |

**Electron 桌面版中的服务管理：**
- 主进程 `main.js` 通过 `child_process.spawn` 启动两个 Python 服务
- `windowsHide: true` 隐藏终端窗口
- 轮询 `/health` 端点等待就绪（500ms 间隔，最多 20 次）
- `before-quit` 事件自动 kill 两个进程

### 3.4 数据流设计

```
用户操作
    │
    ├── AI 翻译 ──► callAI() ──► fetch() POST ──► 火山引擎 API
    │                    └──────── 返回文本 ──────────┘
    │
    ├── 语音播放 ──► speakText() ──► Edge-TTS ──► 失败 → 讯飞 ──► 失败 → Web Speech

    │
    ├── 历史记录 ──► saveHistory()

    │                    └── dbFetch() /api/history POST（异步，静默）
    │
    └── 评测结果 ──► syncEvalToBackend()

```

**双写策略设计原则**：
- localStorage 始终优先响应，保证 UI 无延迟
- 后端写入使用 `fetch().catch(()=>{})` 静默处理错误，后端不可用不影响正常功能
- 前端不依赖后端数据加载，离线可用

## 四、核心实现技术

### 4.1 大模型零样本方言翻译

**问题**：关中、陕北、陕南三大方言区缺乏大规模平行语料，传统 NMT 无法训练。

**方案**：利用大语言模型对汉语方言的既有知识，通过角色扮演 System Prompt 激活方言生成能力。

**三方言差异化 Prompt 设计：**

| 方言区 | 角色设定 | 语气风格 | 特征替换示例 | max_tokens | temperature |
|--------|---------|---------|-------------|------------|-------------|
| 关中 | 老西安本地人 | 市井口语，句式简短 | 吃→咥、聊天→谝闲传、很好→嫽咋咧 | 150 | 0.2 |
| 陕北 | 黄土高原老乡 | 豪迈质朴，语言直白 | 舒服→舒坦、走吧→走哇、可好了→可美咧 | 150 | 0.2 |
| 陕南 | 陕南本地居民 | 轻柔温婉，语气舒缓 | 挺舒服→蛮舒服、知道吗→晓得不 | 150 | 0.2 |

**API 调用：**
- 端点：`https://ark.cn-beijing.volces.com/api/v3/chat/completions`（OpenAI 兼容格式）
- 模型：豆包端点模型
- 鉴权：Bearer Token

### 4.2 方言特征词密度自动评估算法

在不依赖人工标注的条件下，量化方言翻译质量。

**算法流程：**

```
输入: 翻译文本 text, 目标方言区 dialect ∈ {guanzhong, shanbei, shannan}
输出: 评分 score ∈ [0, 98]

Step 1 — 加载特征词集合 F_d:
    F_guanzhong = {咥, 嫽咋咧, 谝, 咋咧, 受活, 碎娃, 额, 木乱, 克里马擦, 哈怂, 扎势, 麻达}  (12个)
    F_shanbei   = {敞亮, 舒坦, 走哇, 可美咧, 好活, 咋接, 甚, 哩}                      (8个)
    F_shannan   = {蛮舒服, 可好啦, 晓得不, 得行, 莫问题, 蛮, 嘛}                       (7个)

Step 2 — 统计命中:
    for each w_i ∈ F_d:
        if text.includes(w_i):
            found += 1

Step 3 — 计算分数:
    base_score = 60
    bonus = min(40, found × 10)
    score = min(98, base_score + bonus)

Step 4 — 评级映射:
    if score ≥ 85:  grade = "优秀" (含 ≥3 个特征词)
    if 70 ≤ score < 85: grade = "合格" (含 1-2 个特征词)
    if score < 70: grade = "不足" (未检出)
```

**评测语料库**：6 场景 × 50 条，Fisher-Yates 随机抽样，单轮产生 150 个数据点，数据持久化至 localStorage 和 SQLite。

### 4.3 三级异构语音合成与透明级联容错

**引擎路由决策树：**

```
speakText(text, style, btn)
    │
    ├── 引擎 = 'edgetts' ──► speakWithEdgeTTS()
    │       │
    │       ├── 成功 → 播放 MP3 → 结束
    │       └── 失败 → speakWithWebSpeech() → 结束

    │       │
    │       ├── 成功 → 播放 MP3 → 结束
    │       └── 失败 → speakWithWebSpeech() → 结束
```

**Edge-TTS 方言参数映射：**

| Style | 语速 | 音调 | 效果 |
|-------|------|------|------|
| `guanzhong` | +8% | +0Hz | 关中市区口语偏快节奏 |
| `shanbei` | -8% | -2Hz | 陕北黄土高原豪迈低缓 |
| `shannan` | -3% | +3Hz | 陕南温婉柔和 |
| `classical` | -25% | -4Hz | 古典雅言缓诵仪式感 |

**服务端容错：** 指数退避重试（0.8×2^n + uniform(0, 0.5)），最多 3 次，文件大小校验 ≥100 bytes 防止空内容。

### 4.4 前后端双重 TTS 文本预处理

**JS 端（前端）：**
```
1. trim + 去多余空白 → 补句末标点
2. 语气词正则替换: /([咧呢嘛哈呀哇哦咯哩呗噢啊])([^\s，,。！？!?…—])/g → '$1，$2'
3. 长句拆分: 按 18 字阈值 + 动词/方位词位置插入逗号
```

**Python 端（服务端）：**
```python
def make_text_speakable(text):
    text = re.sub(r'\s+', '', text.strip())
    text += '。'  # 句末补齐
    # 语气词后加逗号
    text = re.sub(r'([咧呢嘛哈呀哇哦咯哩呗噢啊])([^\s，,。！？!?…—\-])', r'\1，\2', text)
    # 连词前加逗号
    text = re.sub(r'([^，,。！？])(但是|可是|不过|然而|所以|而且|然后|那|就|还)', r'\1，\2', text)
    return text
```

**效果对比：**

| 原始 | 处理后 |
|------|--------|
| 这地方嫽咋咧你啥时候来 | 这地方嫽咋咧，你啥时候来。 |
| 走走走咱到回民街咥羊肉泡馍去 | 走走走，咱到回民街咥羊肉泡馍去。 |

### 4.5 纯前端 HMAC-SHA256 签名算法

**背景**：讯飞 TTS WebSocket API 需要 HMAC-SHA256 签名，而浏览器 `SubtleCrypto` 在 `file://` 协议下不可用。需纯 JS 实现完整 SHA-256 + HMAC。

**算法组成：**
- 64 个 SHA-256 常量 K[t]（NIST FIPS 180-4）
- 6 个逻辑函数：Ch, Maj, Σ₀, Σ₁, σ₀, σ₁
- 消息填充：0x80 追加 → 零填充至 448 mod 512 → 64-bit 位长度追加
- 64 轮压缩迭代，8 个 32-bit 工作变量
- HMAC 标准构建：H((K⊕opad) ‖ H((K⊕ipad) ‖ M))

**UTF-8 多字节编码**：在 `toBytes()` 中完整实现 1-4 字节 UTF-8 序列编码，确保中文与其他平台哈希一致。

**约 20 行核心逻辑，零外部依赖。**

### 4.6 Web Audio API 过程化音效合成

4 种交互音效全部代码合成，无外部音频文件：

| 音效 | 振荡器 | 频率 | 包络 |
|------|--------|------|------|
| 咔嗒（tick） | Triangle | 800Hz | 40ms，增益 0.08→0 |
| 铜锣（gong） | Sine | 520→200Hz 指数滑降 | 700ms，增益 0.15→0 |
| 签筒（rattle） | 6×Square | 300-900Hz 随机 | 间隔 40ms 触发，各 30ms |
| 官印（stamp） | Sine + Triangle 叠加 | 120→40Hz / 60Hz | 主 300ms + 辅 200ms |

### 4.7 SQLite 数据持久化

| `settings` | key (PK), value, updated_at | — |
| `dialect_vocabulary` | id, word (UNIQUE), pronunciation, meaning, region, allusion | region |

## 五、创新点

### 创新点一：低资源条件下的零样本方言翻译与可量化评测闭环

**解决的问题**：方言翻译缺乏大规模平行语料，传统方法不可行。翻译质量缺乏客观评价标准。

**创新之处**：
- 不依赖平行语料微调，通过角色扮演 System Prompt 激活 LLM 方言知识
- 自建 27 个方言特征词库 + 特征词密度自动化评分算法，将主观方言翻译质量转为可比较的客观分数
- 建立 6 场景 50 条标准语料库，形成"翻译 → 评分 → 对比分析"的完整评测闭环

### 创新点二："三级引擎 + 透明降级"的全场景语音合成容错架构

**解决的问题**：单一方言语音引擎在比赛演示中不可靠（网络波动、服务限流、环境不同）。

**创新之处**：
- Edge-TTS（主力，陕西话神经语音）→ 讯飞 WebSocket（备选）→ Web Speech（离线兜底）三级级联
- 各引擎故障自动检测与切换，用户无感知
- 五档语速/音调参数差异化模拟三种方言韵律
- 纯前端 HMAC-SHA256 签名突破 file:// 协议限制，零部署依赖

### 创新点三：面向 TTS 自然度增强的前后端双重文本预处理

**解决的问题**：TTS 引擎对无标点长句"连读不换气"，方言语气词被平滑掉。

**创新之处**：
- 12 种方言语气词的自动逗号插入，保留方言韵律特征
- 长句智能断句（18 字阈值 + 动词/方位词位置），模拟自然说话节奏
- 前端 JS 和后端 Python 双重预处理，覆盖直接调用 API 和本地服务两种路径

### 创新点四：AI + 游戏化 + 非遗保护的"体验式传承"融合范式

**解决的问题**：传统方言保护以静态存档为主（录音、文献），公众参与度低。

**创新之处**：
- 构建覆盖"工具层（翻译/解码）→ 教育层（考试）→ 娱乐层（祈福/转盘）→ 展示层（保护看板）"的完整体验系统
- 将方言从"被保护的客体"变为"可玩的文创内容"，降低年轻用户体验门槛
- 盛唐美学贯穿全平台，使文化 IP 与 AI 技术形成有机整体
