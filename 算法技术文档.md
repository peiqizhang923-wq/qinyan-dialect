## 二、面向低资源方言场景的大语言模型翻译方法

### 2.1 问题定义

给定标准汉语输入文本 $T$ 和目标方言区 $d \in \{\text{关中}, \text{陕北}, \text{陕南}\}$，任务是在缺乏大规模平行语料的条件下，生成符合该方言区语音、词汇、句法特征的翻译结果 $\hat{T}_d$。

### 2.2 基于角色扮演提示词的零样本翻译

本方法不依赖标注平行语料进行微调，而是利用大语言模型对汉语方言的既有知识，通过精确设计的 System Prompt 激活模型对方言特征的理解。

**三方言差异化提示词设计：**

| 方言区 | 提示词策略 | 特征词汇示例 |
|--------|-----------|-------------|
| **关中** | 扮演老西安本地人，市井口语风格，句式简短接地气 | 咥(吃)、谝闲传(聊天)、嫽咋咧(很好)、咋咧(怎么了)、受活(舒服) |
| **陕北** | 扮演黄土高原本地老乡，豪迈质朴风格，语言直白爽朗 | 敞亮、舒坦、走哇、可美咧、好活 |
| **陕南** | 扮演陕南本地居民，轻柔温婉风格，语气舒缓柔和 | 蛮舒服、可好啦、晓得不、得行、莫问题 |

**API 调用参数配置：**

| 参数 | 值 | 说明 |
|------|-----|------|
| `max_tokens` | 150 | 方言转换无需长文本，限制输出长度保证响应速度 |
| `temperature` | 0.2 | 低温度保证翻译结果的一致性与可控性，减少随机发散 |
| API 端点 | `https://ark.cn-beijing.volces.com/api/v3/chat/completions` | 火山引擎豆包大模型（OpenAI 兼容接口） |
| 模型 | `ep-20260521204922-j5x8l` | 豆包端点模型 |

### 2.3 方言特征词密度自动评估算法

在不借助人工标注的情况下，通过统计翻译结果中方言特征词的出现密度来量化翻译质量。

**算法流程：**

```
输入: 翻译文本 text, 目标方言区 dialect
输出: 评分 score ∈ [0, 98]

1. 加载方言 d 的特征词集合 F_d
2. 遍历 F_d 中每个特征词 w_i:
     如果 w_i 在 text 中出现:
         found += 1
3. base_score = 60
4. bonus = min(40, found × 10)
5. score = base_score + bonus
6. 返回 min(98, score)
```

**三方特征词库：**

| 方言区 | 特征词数量 | 特征词 |
|--------|-----------|--------|
| 关中 | 12 | 咥、嫽咋咧、谝、咋咧、受活、碎娃、额、木乱、克里马擦、哈怂、扎势、麻达 |
| 陕北 | 8 | 敞亮、舒坦、走哇、可美咧、好活、咋接、甚、哩 |
| 陕南 | 7 | 蛮舒服、可好啦、晓得不、得行、莫问题、蛮、嘛 |

**评分区间映射：**

| 分数区间 | 评级 | 含义 |
|----------|------|------|
| ≥ 85 | 高 | 转译质量优秀，含 ≥ 3 个方言特征词 |
| 70–84 | 中 | 转译质量合格，含 1–2 个方言特征词 |
| < 70 | 低 | 转译质量不足，未检出方言特征词 |

### 2.4 评测语料库设计

建立覆盖 **6 大场景类别、50 条标准测试句** 的评测语料库，每条语料跨三种方言评测，单轮完整评测产生 150 个数据点。

| 类别 | 条目数 | 示例 |
|------|--------|------|
| 日常问候 | 8 | "你好，今天天气真好"、"吃了吗，最近怎么样" |
| 饮食文化 | 10 | "来一碗羊肉泡馍"、"肉夹馍要肥瘦的才香" |
| 天气环境 | 8 | "今天风很大，出门多穿点"、"秋高气爽，适合出游" |
| 情感表达 | 8 | "我今天特别开心"、"别担心，一切都会好的" |
| 动作行为 | 7 | "快走，要迟到了"、"坐下来慢慢吃" |
| 地点场景 | 7 | "城墙上面人真多"、"回民街的小吃真多" |

评测时采用 **Fisher-Yates 随机抽样**，支持 10 条（30次翻译）和 25 条（75次翻译）两种规模。所有结果持久化至 localStorage，支持跨会话的累计统计。

## 三、三引擎异构方言语音合成与级联容错机制

### 3.1 架构总览

```
speakText(text, style, btn)
    │
    ├── 引擎选择: getVoiceEngine()
    │
    ├── [edgetts] ──► speakWithEdgeTTS() ──┐
    │       │                               │
    │       └── 失败 ──► speakWithWebSpeech()│
    │                                       │
    ├── [xunfei] ───► speakWithXunfei() ───┤
    │       │                               │
    │       └── 失败 ──► speakWithWebSpeech()│
    │                                       │
    └── [browser] ──► speakWithWebSpeech() ─┘
```

引擎选择优先级：**Edge-TTS > 讯飞 > 浏览器语音**。任意引擎失败时自动降级至浏览器语音，降级过程对用户透明。

### 3.2 Edge-TTS — 主力引擎

调用微软 Azure 神经网络语音 `zh-CN-shaanxi-XiaoniNeural`，该语音是微软官方发布的陕西话语音模型。

**方言风格参数映射：**

| 风格参数 | 语速 (rate) | 音调 (pitch) | 设计意图 |
|----------|-------------|-------------|----------|
| `guanzhong` | +8% | +0Hz | 关中话偏快节奏，对应市区口语习惯 |
| `shanbei` | -8% | -2Hz | 陕北话偏缓，低沉豪迈感 |
| `shannan` | -3% | +3Hz | 陕南话柔和，微升音调体现温婉 |
| `classical` | -25% | -4Hz | 古风诵读，显著减速营造仪式感 |
| `normal` | +0% | +0Hz | 标准语速，通用场景 |

**服务端重试与容错：**

```
FOR attempt = 1 TO 3:
    1. 调用 edge_tts.Communicate() 合成音频
    2. 检查输出文件大小 >= 100 bytes（防止空内容）
    3. 成功 → 返回音频文件
    4. 失败 → 等待 delay = 0.8 × 2^attempt + random(0, 0.5) 秒后重试
THROW 重试3次后仍失败
```

采用指数退避 + 随机抖动的重试策略，应对微软免费接口的偶发限流与网络波动。服务端为多线程 HTTP（`ThreadingMixIn`），支持并发 TTS 请求。

### 3.3 讯飞 TTS — 副引擎（纯前端 WebSocket 鉴权）

讯飞流式 TTS 需要 HMAC-SHA256 签名鉴权。为实现 `file://` 协议下的纯前端运行，在 `common.js` 中完整实现了 SHA-256 哈希与 HMAC 算法（约 20 行，零依赖）。

**鉴权流程：**

```
1. 生成 RFC 1123 格式 UTC 时间: date = new Date().toUTCString()
2. 构建签名字符串:
   signature_raw = "host: tts-api.xfyun.cn\ndate: {date}\nGET /v2/tts HTTP/1.1"
3. HMAC-SHA256(signature_raw, apiSecret) → Base64 编码 → signature
4. 构建鉴权头:
   Authorization = api_key="{apiKey}", algorithm="hmac-sha256",
                   headers="host date request-line", signature="{sig}"
5. 建立 WebSocket 连接:
   wss://tts-api.xfyun.cn/v2/tts?authorization={auth}&date={date}&host=tts-api.xfyun.cn
6. 发送合成参数:
   { common: {app_id}, business: {aue:'lame', vcn:voice_id, speed:50, pitch:50},
     data: {status:2, text:base64(utf8_text)} }
7. 接收流式音频帧 → 合并 → 转为 Blob → 播放
```

WebSocket 15 秒超时自动断开，防止挂起。讯飞不可用时自动降级至 Web Speech。

### 3.4 Web Speech API — 兜底引擎

浏览器内置 `SpeechSynthesis` API，无需网络、无需额外依赖。

**方言参数模拟：**

| 风格 | rate | pitch | 效果 |
|------|------|-------|------|
| 关中 | 1.15 | 1.15 | 语速稍快，音调偏高 |
| 陕北 | 0.85 | 0.85 | 语速偏慢，音调偏低 |
| 陕南 | 0.95 | 1.05 | 语速略慢，音调微升 |
| 古风 | 0.70 | 0.95 | 显著减速诵读 |

虽无法复现 Edge-TTS "陕西话 Xiaoni" 的纯正方言音色，但通过语速和音调参数调节，能在一定程度上区分三种方言的风格差异。

## 四、面向 TTS 自然的文本预处理算法

### 4.1 算法设计动机

Edge-TTS 等现代 TTS 引擎在朗读无标点长句时容易出现"连读不换气"的问题，影响听觉自然度。陕西方言中存在大量语气助词（咧、呢、嘛、哈等），这些词是方言韵律的关键标志，需要为 TTS 引擎提供明确的停顿提示。

### 4.2 预处理流程

```
输入: 原始文本 text
输出: 优化后的可朗读文本

步骤1 — 空白规范化:
    text = trim(text).replace(/\s+/g, '')

步骤2 — 句末标点补齐:
    IF text 不以 。！？!? 结尾:
        text += '。'

步骤3 — 语气词后插入逗号停顿:
    pattern = /([咧呢嘛哈呀哇哦咯哩呗噢啊])([^\s，,。！？!?…—])/g
    text = text.replace(pattern, '$1，$2')

步骤4 — 长句拆分 (>18字且无标点):
    FOR 每个以，,。！？!? 分隔的子句 p:
        IF p.length > 18 AND p 中无逗号:
            在动词/方位词(去走来上到在给把被让跟和与)前 8-12 字处插入逗号
```

**示例：**

| 原始文本 | 预处理后 |
|----------|---------|
| `这地方嫽咋咧你啥时候来` | `这地方嫽咋咧，你啥时候来。` |
| `走走走咱到回民街咥羊肉泡馍去` | `走走走，咱到回民街咥羊肉泡馍去。` |

### 4.3 服务端二次预处理

Python TTS 服务端在收到请求后执行额外的规范化处理：

```python
def make_text_speakable(text):
    # 1. 去空白、补句末标点
    text = re.sub(r'\s+', '', text.strip())
    if not re.search(r'[。！？!?~～…—」』\)]$', text):
        text += '。'
    # 2. 语气词 + 逗号
    text = re.sub(r'([咧呢嘛哈呀哇哦咯哩呗噢啊])([^\s，,。！？!?…—\-])', r'\1，\2', text)
    # 3. 连词前加逗号（但是、可是、不过、然而、所以、而且、然后、那、就、还）
    text = re.sub(r'([^，,。！？])(但是|可是|不过|然而|所以|而且|然后|那|就|还)', r'\1，\2', text)
    return text
```

前后端双重预处理确保不同调用路径下都能获得自然的 TTS 输出。

## 五、Web Audio API 过程化音效合成

为避免依赖外部音频文件，系统所有交互音效均通过 Web Audio API 实时合成。

### 5.1 音效设计

| 音效 | 触发场景 | 合成方式 | 参数 |
|------|---------|---------|------|
| **咔嗒声** (tick) | 转盘旋转每 240ms | Triangle 振荡器，800Hz | 40ms 衰减包络，增益 0.08→0 |
| **铜锣声** (gong) | 转盘停止 | Sine 振荡器，520→200Hz 指数频率滑降 | 700ms 衰减，增益 0.15→0 |
| **签筒声** (rattle) | 抽签摇晃 | 6 个 Square 振荡器随机频率(300-900Hz) | 间隔 40ms 依次触发，各 30ms |
| **官印声** (stamp) | 雅言印章落下 | Sine 120→40Hz (主) + Triangle 60Hz (叠加) | 主 300ms + 辅 200ms，模拟混响 |

### 5.2 关键技术要点

- **惰性初始化**：`AudioContext` 仅在首次调用音效时创建，避免浏览器自动播放策略限制
- **短生命周期**：每个振荡器在触发后立即 `start()` 并 `stop()`，不保留长驻节点
- **指数衰减**：使用 `exponentialRampToValueAtTime()` 实现自然的衰减包络，避免线性衰减的机械感

## 六、HMAC-SHA256 纯前端实现

### 6.1 实现背景

讯飞 TTS WebSocket API 要求 HMAC-SHA256 签名，且项目需在 `file://` 协议下运行。标准方案需依赖 Node.js `crypto` 模块或浏览器 `SubtleCrypto`（后者要求 HTTPS 或 localhost，在 `file://` 下不可用）。因此实现了完整的纯 JavaScript SHA-256 + HMAC 算法。

### 6.2 算法实现

**SHA-256 核心：**
- 64 个常量 $K_t$（NIST FIPS 180-4 定义）
- 6 个逻辑函数：$Ch$, $Maj$, $\Sigma_0$, $\Sigma_1$, $\sigma_0$, $\sigma_1$
- 消息填充：`0x80` 追加 → 零填充至 $448 \bmod 512$ → 64-bit 位长度追加
- 64 轮压缩迭代，8 个 32-bit 工作变量

**HMAC 构建：**
$$ \text{HMAC}(K, M) = H\left((K \oplus \text{opad}) \parallel H\left((K \oplus \text{ipad}) \parallel M\right)\right) $$

其中 $\text{opad} = \text{0x5c5c}...$，$\text{ipad} = \text{0x3636}...$，各 64 字节。密钥长于 64 字节时先 SHA-256 哈希再填充。

**API 暴露：**
```javascript
XF_HMAC.hmac(key, message)         // → hex string
XF_HMAC.hmacBase64(key, message)   // → base64 string
XF_HMAC.sha256(message)            // → hex string
```

### 6.3 Unicode 处理

中文字符的 UTF-8 编码在 `toBytes()` 函数中完整实现多字节序列编码（1-4 字节），确保中文字符哈希值与非中文环境的 Python/CryptoJS 计算结果一致。

## 七、Electron 桌面封装中的子进程管理

### 7.1 Python TTS 服务生命周期

```
App 启动
  │
  ├── spawn('python', ['tts_server.py'], {windowsHide: true})
  │     │
  │     ├── 轮询 GET /health (500ms 间隔, 最多 20 次)
  │     │     ├── 200 → 服务就绪, 创建窗口
  │     │     └── 超时 → 放弃等待, 仍创建窗口(TTS降级)
  │     │
  │     └── pythonProc.on('error') → 记录, 不阻塞
  │
  ├── BrowserWindow 加载 index.html
  │
App 退出
  │
  ├── before-quit: pythonProc.kill()
  └── window-all-closed: pythonProc.kill()
```

### 7.2 关键设计决策

| 决策 | 理由 |
|------|------|
| `windowsHide: true` | 隐藏 Python 终端窗口，保持桌面应用的干净体验 |
| `nodeIntegration: false` | 安全最佳实践，渲染进程不接触 Node.js API |
| `contextIsolation: true` | 预加载脚本与页面上下文隔离 |
| Python 不可用不阻塞启动 | 应用核心功能（AI翻译）不依赖 Python，TTS 自动降级 |

## 九、主要创新点

1. **低资源条件下的零样本方言翻译策略**：不依赖平行语料，通过角色扮演提示词 + 特征词密度评估，实现三种方言的可用品质翻译，并建立了可量化的自动化评测框架。

2. **前后端协同的 TTS 自然度增强管线**：前端 JavaScript 和后端 Python 分别对文本进行模态词停顿插入预处理，配合 Edge-TTS 语速/音调差异化参数，显著提升合成语音的方言韵律感。

3. **全场景降级容错的异构语音合成架构**：Edge-TTS（本地服务）→ 讯飞（WebSocket 纯前端鉴权）→ Web Speech（离线兜底）三级级联，任意链路故障自动切换，HMAC-SHA256 签名无需后端代理即可完成。
